# 実物オブジェクト追跡の実装方針

## 目的

水平ディスプレイ上に置かれた、事前登録済みの実物オブジェクトを上方カメラで検出し、既存のp5.jsシミュレーションへ餌または障害物として反映する。鑑賞者から認識用マーカーが見えないこと、物体数を固定しないこと、展示中にカメラが一時停止しても作品全体は止まらないことを要件とする。

## 採用する方式

カメラ映像に対する専用のインスタンスセグメンテーションを採用する。

- 最終展示で使う個体を、実際の朽木背景・照明・カメラ位置で撮影する
- 学習クラスは`food`、`obstacle`とする。手は負例として学習し、必要なら補助クラス`hand`を加える
- マスクから中心、輪郭、大きさを取得し、フレーム間追跡で一時的なIDを安定させる
- 学習時に想定しない物体は、確信度が十分でない限り無視する
- 物体が変更された場合は、展示前にデータ追加と再学習を行う

一般的なキノコや石を認識する汎用モデルを目指すのではなく、こちらで用意した実物と展示環境に限定して精度を上げる。見えるArUcoマーカーは採用しない。RGBだけで必要精度を満たせない場合に限り、深度カメラなどを代替案として再検討する。

## システム構成

```text
iPad Air（上方約70cm、背面カメラ）
    ↓ USB-C / Camo Camera
Camo Studio（露出・WB・フォーカス固定）
    ↓ 仮想カメラ映像
追跡サーバー（Python）
    ├─ インスタンスセグメンテーション
    ├─ フレーム間追跡・平滑化・欠落猶予
    ├─ 画面座標への変換
    └─ 状態・ログの管理
    ↓ WebSocket / JSON（目標10Hz）
ブラウザ（p5.js、描画30fps）
    ├─ 動的な餌・障害物一覧を更新
    ├─ 誘引場・障害物マスクを更新
    ├─ 1日分の状態を定期保存
    └─ 展示映像だけを描画
```

本番はM1 Mac mini上で追跡サーバーとブラウザを同時に動かす。学習や開発にはMacBook Airも使用できる。追跡と描画を別プロセスにし、カメラやモデルを使わないモック入力でも統合動作を確認できるようにする。

## カメラと画面座標

- iPad Airは画面中央の真上、保護板から約70cmの位置へ固定する
- タブレット用クランプに加えて落下防止ワイヤーを付ける
- 左右から拡散した固定照明を当てる
- 自動露出、自動ホワイトバランス、自動フォーカスは展示条件に合わせて固定する
- Camoの映像をOpenCVから取得できない場合は、OBS Virtual Cameraを中継する

設置後にカメラ映像上の画面四隅を運営者が指定し、その4点からホモグラフィを計算して保存する。追跡結果の中心と輪郭に同じ変換を適用し、画面左上を`(0, 0)`、右下を`(1, 1)`とする正規化座標へ変換する。カメラや箱を動かしたときだけ再キャリブレーションする。

物体の高さによる視差は実データで評価する。誤差が大きい場合は物体ごとの高さ補正を設定へ追加する。

## 通信形式

追跡サーバーは、確定済み物体の完全な一覧を一定間隔で送る。

```json
{
  "type": "tracking-frame",
  "timestamp": 1784250000000,
  "sequence": 1204,
  "camera": "ok",
  "objects": [
    {
      "id": "track-7",
      "kind": "food",
      "x": 0.31,
      "y": 0.62,
      "confidence": 0.94,
      "moving": false,
      "contour": [[0.28, 0.58], [0.34, 0.59], [0.35, 0.65], [0.29, 0.66]]
    },
    {
      "id": "track-12",
      "kind": "obstacle",
      "x": 0.68,
      "y": 0.41,
      "confidence": 0.91,
      "moving": true,
      "contour": [[0.61, 0.37], [0.74, 0.36], [0.76, 0.45], [0.62, 0.46]]
    }
  ]
}
```

- `objects`は0件以上で上限をプロトコル上に設けない
- `id`は追跡中の同一性を保つ一時IDとする
- `kind`は`food`または`obstacle`だけをブラウザへ渡す
- `x`、`y`、`contour`は0〜1へ正規化する
- `contour`は簡略化した閉多角形とし、障害物の当たり判定にも使う
- `moving`は平滑化後の速度とデッドゾーンから追跡サーバーが決定する
- `camera`が`lost`のフレームでは、最後に安定した物体一覧を保持する

## 安定化

- 指数移動平均で中心と輪郭を平滑化する
- 小さな移動はデッドゾーンで無視する
- 新規検出は複数フレーム続いた後に追加する
- 欠落後0.5秒は最後の位置を保持する
- 削除も複数フレームまたは欠落猶予の経過後に確定する
- 手で覆われた時間は既存物体の欠落として扱い、手そのものは送信しない
- カメラ切断時は最後の確定状態を凍結し、運営用ステータスとログだけを更新する

閾値は設定ファイルに置き、学習データや展示条件を変えずに調整・比較できるようにする。

## シミュレーション側

### 動的な物体数

追跡フレームを完全な物体一覧として扱い、同じIDの状態を更新し、新しいIDを追加し、消えたIDを削除する。固定個数は持たない。マウス入力を使う開発モードだけは従来の初期個数を既定値として残す。

### 餌

- 同じIDなら座標が変わっても栄養量と包囲状態を維持する
- 動かし終えた餌を補充済みとして扱う現在の挙動は、ひとまず維持する
- 餌が0個でも例外を起こさない
- 餌がない間は粒子数、明るさ、移動速度を徐々に下げて休眠させる
- 餌が戻ると徐々に通常状態へ戻す

### 障害物

- 輪郭ポリゴンを当たり判定に使う
- 位置または輪郭の変化が閾値を超えたときだけ障害物マスクを再構築する
- マスク更新時は重なった粒子を安全な位置へ移す
- 検出揺れだけで粒子が押し出され続けないようにする

### 表示と状態保存

- 展示モードでは餌、障害物、文字、UI、カーソル、認識結果を描画しない
- デバッグモードではマウス操作と認識状態の確認を許可する
- シミュレーション状態をブラウザ内へ定期保存し、同じ展示日のクラッシュ後に復元する
- 日付が変わった場合は新規状態から開始する
- rawカメラ映像は本番運用中に保存しない

## 実装順

1. 通信スキーマ、設定、モック追跡サーバーを実装する
2. p5.jsを0〜N個の追跡物体、展示表示、休眠へ対応させる
3. カメラ入力とキャリブレーションを実装する
4. データ撮影、アノテーション、学習、評価の補助コマンドを用意する
5. 実物決定後に専用データセットを作り、モデルを学習する
6. 最終の箱・照明・カメラ位置で精度と10時間稼働を検証する

## 最初の完了条件

- モック入力で餌・障害物が0〜N個追加、移動、削除される
- 餌0個の状態で生命体が消えずに休眠する
- 追跡停止時に最後の配置でシミュレーションが継続する
- 展示モードの画面にシミュレーション以外が一切出ない
- 実カメラで登録物体と未登録物体を区別できる
- 画面短辺に対する中心位置誤差が1%程度以内である
- 操作から反映までの遅延が150ms程度以内である
- 空の画面や手だけの場面で物体が誤追加されない
- 30分試験後、最終的には10時間の連続動作で追跡が大きくずれない

モデル精度の判定方法とデータ分割は[物体認識の実験計画](tracking-experiment.md)に定義する。

## 実装前に残る外部条件

ソフトウェアの骨格は現在の情報で実装できる。実物を使う最終調整には、次の確定・確認が必要である。

- 使用する餌・障害物の個体と寸法
- Dellディスプレイの型番、実解像度、映像端子、排気位置
- iPad Airの世代と、Camo映像がOpenCVから取得できるか
- 最終的な箱と照明を組んだ状態での撮影データ
