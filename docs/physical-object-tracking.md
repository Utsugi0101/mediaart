# 実物オブジェクト追跡の実装方針

## 目的

現在は画面上でドラッグしている「餌」と「障害物」を、ディスプレイ上に置かれた実物のキノコと石へ置き換える。

鑑賞者が実物を動かすと、その位置がシミュレーションへ反映され、生命体の誘引方向や経路が変化する状態を目指す。実物が画面上の存在そのものになるため、展示モードでは餌や障害物の図形を原則として描画しない。

## 採用する方式

最初の実装では、水平ディスプレイの上方に固定したカメラと、実物に付けたArUcoマーカーを使用する。

- カメラはディスプレイ全体を真上から撮影する
- キノコと石には、それぞれ異なるIDのマーカーを付ける
- OpenCVでマーカーのID、中心位置、回転角度を検出する
- 検出結果をディスプレイ上の正規化座標へ変換する
- 変換した情報を既存のp5.jsシミュレーションへ送信する

色や形だけを使った認識は、照明、影、手、画面に表示される映像の影響を受けやすいため、最初の方式には採用しない。まずマーカー方式で体験全体を成立させ、その後にマーカーを造形へなじませる方法や、赤外線を使った認識などを検討する。

## システム構成

```text
上方カメラ
    ↓ 映像
物体追跡プロセス（Python + OpenCV）
    ├─ マーカーのID・中心・角度を検出
    ├─ カメラ座標を画面座標へ変換
    └─ 揺れ・一時的な見失いを補正
    ↓ WebSocket / JSON
ブラウザ（p5.js）
    ├─ 餌と障害物の状態を更新
    ├─ 誘引場・障害物マスクを更新
    └─ シミュレーションを描画
```

画像認識とシミュレーションを別プロセスに分ける。これにより、カメラやOpenCVの問題をシミュレーションから切り離して確認でき、展示中のログ取得や再起動もしやすくなる。

現在のマウス操作は削除せず、カメラがない開発環境とデバッグ用の入力として残す。

## 物体とマーカーの対応

マーカーIDから物体の種類とシミュレーション上の大きさを決定する。

初期の割り当て例は次のとおりとする。

| マーカーID | 種類 | 用途 |
| --- | --- | --- |
| 1〜9 | `food` | キノコ型の餌 |
| 10〜29 | `obstacle` | 石型の障害物 |
| 40〜43 | `calibration` | ディスプレイ四隅の基準点 |

IDごとの設定は追跡コードへ直接埋め込まず、設定ファイルで管理できる形を目指す。

```json
{
  "1": {
    "kind": "food",
    "radius": 0.025
  },
  "10": {
    "kind": "obstacle",
    "radiusX": 0.08,
    "radiusY": 0.05
  }
}
```

大きさは画面の短辺に対する比率として持つ。障害物の判定範囲は実物の底面より少し大きくし、生命体が石の縁に重なって見えないようにする。

## 座標変換とキャリブレーション

カメラ画像の座標をそのままシミュレーションへ渡すのではなく、ディスプレイ四隅との対応から透視変換を行う。

1. ディスプレイの四隅に基準マーカーを置く
2. カメラ画像上で四隅の座標を取得する
3. 四隅が`(0, 0)`、`(1, 0)`、`(1, 1)`、`(0, 1)`となる変換行列を計算する
4. 各物体の中心座標へ同じ変換を適用する
5. p5.js側で正規化座標を`gridWidth`と`gridHeight`へ変換する

カメラとディスプレイを固定できる場合、変換行列は保存して再利用する。カメラを動かした場合や展示を組み直した場合は再キャリブレーションする。

マーカーを物体の上面へ付けると、その高さによる視差が発生する。最初はカメラを画面中央の真上に置き、できるだけ距離を取って影響を小さくする。精度が不足する場合は、物体ごとの高さ補正またはカメラの内部・外部パラメーターを使った補正を追加する。

## 通信形式

追跡プロセスは、認識できた物体を一定間隔でブラウザへ送信する。

```json
{
  "type": "tracking-frame",
  "timestamp": 1784250000000,
  "objects": [
    {
      "id": "food-1",
      "markerId": 1,
      "kind": "food",
      "x": 0.31,
      "y": 0.62,
      "angle": 0.0
    },
    {
      "id": "obstacle-10",
      "markerId": 10,
      "kind": "obstacle",
      "x": 0.68,
      "y": 0.41,
      "angle": 1.2
    }
  ]
}
```

- `x`と`y`は0〜1の範囲とする
- `angle`はラジアンとする
- `id`は物体の状態を継続して保持するための安定した識別子とする
- カメラや追跡処理の状態は、物体情報とは別のステータスメッセージで通知する

## シミュレーション側の変更方針

入力方法を「マウス」と「物体追跡」に分離し、どちらも同じ餌・障害物モデルを更新する構成にする。

### 餌

- 同じIDの餌は、座標が変わっても同じ個体として扱う
- 毎フレーム作り直さず、栄養量や包囲状態を維持する
- 一時的に見失っただけの場合は最後の位置を保持する
- 一定時間取り除かれた後に再配置された場合は、新しい餌として栄養量を戻す
- 餌が0個でもシミュレーションが停止・例外発生しないようにする

### 障害物

- 同じIDの障害物は、位置と角度を更新する
- 障害物マスクは毎フレーム無条件に再構築せず、移動量が閾値を超えた場合、または一定周期で更新する
- マスクを更新した後は、障害物と重なった粒子を既存の処理で安全な位置へ移動する
- 追跡中の細かな座標揺れだけで生命体が押し出され続けないようにする

### 描画

- 展示モードでは、現在の「餌」と「障害物」の図形とラベルを非表示にする
- 開発モードでは、認識位置、ID、当たり判定範囲、追跡状態を重ねて表示できるようにする
- カメラ入力がなくても、現在のマウスドラッグ操作で開発できる状態を維持する

## 追跡を安定させる処理

カメラの検出値を直接使用すると画面内の物体が細かく震えるため、以下の処理を入れる。

- 数フレームの移動平均または指数移動平均で座標を平滑化する
- ごく小さな移動を無視するデッドゾーンを設ける
- マーカーを見失っても、短い猶予時間は最後の位置を保持する
- 物体の追加と削除は、複数フレーム連続で状態が確認できてから確定する
- 手で隠れた時間と、実際に取り除かれた状態をタイムアウトで区別する

初期値として、見失いの猶予時間は0.5秒程度を想定する。実際のカメラのフレームレートと鑑賞者の操作を見ながら調整する。

カメラとの通信が完全に途切れた場合は、全オブジェクトを即座に削除しない。最後の安定した状態でシミュレーションを継続し、運営者向けの表示またはログで異常を知らせる。

## 実装段階

### 1. カメラ認識の単体試作

- Webカメラでディスプレイ全体を撮影する
- 紙に印刷した複数のArUcoマーカーを検出する
- ID、中心座標、角度を画面またはログに表示する

### 2. 座標変換と通信

- 四隅の基準点から透視変換を行う
- 座標を0〜1へ正規化する
- WebSocketでブラウザへJSONを送信する
- 通信状態を確認できる簡単なデバッグ表示を追加する

### 3. 既存シミュレーションとの統合

- 追跡された餌を`foods`へ反映する
- 追跡された障害物を`obstacles`へ反映する
- 餌が0個の場合と、物体が追加・削除された場合を安全に処理する
- マウス入力とカメラ入力を切り替えられるようにする

### 4. 実物オブジェクトの試作

- キノコと石の簡易模型を作る
- マーカーの大きさ、カメラの高さ、照明条件を調整する
- 座標の揺れ、手による遮蔽、複数物体の接近を試す
- 実物の底面とシミュレーション上の判定範囲を合わせる

### 5. 展示運用への対応

- 起動時の自動接続と再接続を実装する
- カメラ停止時の安全な動作を確認する
- デバッグ表示を隠した展示モードを用意する
- 長時間動作、再起動、再キャリブレーションを確認する
- マーカーを造形へなじませ、展示全体の外観を整える

## 最初の完了条件

- 餌と障害物を同時に複数認識できる
- 物体を動かすと、画面上の対応位置が追従する
- 位置誤差が画面短辺の1%程度以内に収まる
- 操作から表示への遅延が150ms程度以内に収まる
- 手で一時的に隠しても、物体が不自然に消えたり別IDへ切り替わったりしない
- カメラを切断しても、シミュレーション自体は停止しない
- 30分以上の連続動作で追跡が大きくずれない

精度と遅延の数値は初期目標であり、実際のディスプレイ寸法、カメラ、展示体験に合わせて調整する。

## 未決定事項

- 使用するディスプレイの寸法とタッチ機能の有無
- カメラの機種、解像度、設置高さ、固定方法
- 同時に使用する餌と障害物の最大数
- 実物オブジェクトの寸法とマーカーの大きさ
- マーカーを鑑賞者からどの程度見せるか
- 追跡プロセスとブラウザを同じコンピューターで動かすか
- 展示開始時のキャリブレーションを自動化するか、運営者が実行するか
